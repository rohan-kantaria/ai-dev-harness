# AI Development Harness

A reusable Claude Code template that automates the full development lifecycle:
**PRD → Phase Plans → TDD Implementation → Review → Commit**

Built as a capstone project for the Slalom Software Engineering with AI bootcamp. Demonstrates: AI agents, persistent memory, slash commands, context engineering, and Test-Driven Development.

## How to Use This Template

### 1. Fork and Set Up
1. Fork this repo on GitHub
2. Clone your fork locally
3. Open in Claude Code (`claude` in the project directory)
4. Run `/init-project` — answers 7 questions, fills in CLAUDE.md and memory files
5. Run `/prime` — verify context loads correctly

### 2. Plan Your App
6. Run `/create-prd` — structured Q&A generates a Product Requirements Document
7. Run `/plan-feature phase-1-[feature-name]` — creates a step-by-step implementation plan

### 3. Build with TDD
8. Run `/execute .agents/plans/phase-1-[name].md` — implements each step automatically with TDD
9. Run `/review` — validates the phase before advancing
10. Repeat steps 7–9 for each phase

## Command Reference

| Command | What it does | When to use it |
|---|---|---|
| `/help` | List all commands | Any time you forget a command |
| `/init-project` | One-time setup after forking — fills CLAUDE.md and memory files | Once, right after forking |
| `/prime` | Load project context (CLAUDE.md, memory, PRD, active plan) | Start of every session |
| `/create-prd` | Interview mode → writes `.claude/PRD.md` | When planning a new app or feature |
| `/plan-feature [name]` | Analyzes codebase via parallel subagents → writes phase plan | After PRD is ready |
| `/execute [plan-path]` | Full TDD cycle per step: tests → implement → lint → commit | When implementing a phase |
| `/test` | Run tests, fix failures, report | Spot-checking outside of /execute |
| `/commit` | Semantic git commit following your project's style | Any change you want to save |
| `/review` | Phase review: tests, lint, PRD alignment, pass/fail verdict | After /execute finishes a phase |

> **Session rhythm:** `/prime` → work → `/review` → `/prime` (next session)

## Memory System

Memory files persist across Claude sessions. They are **not** auto-loaded — `/prime` reads the index and loads files on demand. This keeps context lean.

| File | What it stores | Written by |
|---|---|---|
| `user_profile.md` | Your role, tech background, collaboration preferences | `/init-project`, you |
| `project_decisions.md` | Key technical decisions + rationale | Agents during `/execute` |
| `feedback_corrections.md` | Session learnings — what worked, what didn't | You (after sessions) |
| `preferences/testing_style.md` | Test framework, TDD conventions, fixture patterns | `/init-project` |
| `preferences/coding_style.md` | Naming, file structure, error handling | `/init-project` |
| `preferences/commit_style.md` | Commit message format | `/init-project` |

## The `/execute` TDD Pipeline

Each step in a phase plan goes through this cycle automatically:

```
1. Classify step type (test / implement / lint / commit / other)
2. WRITE TESTS: create test file → write failing test → run to confirm failure
3. IMPLEMENT: write code → run full suite → fix if failing (max 3 attempts)
4. LINT: run formatter/linter → fix → re-run
5. COMMIT: read commit style → stage files → commit with semantic message
6. Mark step complete in plan file (checkbox: - [ ] → - [x])
7. Advance to next step — or halt and report if unresolvable
```

The command halts and asks for help if:
- Tests fail after 3 fix attempts
- A test passes unexpectedly before the implementation is written
- Lint errors cannot be auto-fixed
