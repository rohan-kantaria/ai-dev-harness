# AI Development Harness — Design Spec
**Date:** 2026-04-14  
**Author:** Rohan Kantaria  
**Context:** Slalom Software Engineering with AI Bootcamp — Capstone Project

## Overview

This project is a **reusable AI development harness** for Claude Code. It is designed to be:
- Hosted on GitHub as a template repository
- Forked to bootstrap any new project with a full AI-assisted development setup
- A live demonstration of bootcamp learnings: AI agents, persistent memory, slash commands, context engineering, and test-driven development

The harness is the deliverable. An app is built *using* the harness as part of the demo — showing the process in action — but the harness itself is the capstone project.

---

## Goals

1. **AI Agents** — subagent delegation for planning, testing, and review tasks
2. **Persistent Memory** — structured memory files that survive across Claude sessions
3. **Slash Commands** — 9 custom commands that automate the full development lifecycle
4. **Test-Driven Development** — `/execute` runs a full TDD cycle (tests first, then implementation, then lint, then commit)
5. **Context Engineering** — CLAUDE.md + PRD + phase plans keep agents focused without polluting conversation context
6. **Reusability** — fork the repo, run `/init-project`, and you have a full AI dev setup for any new project

## Slash Commands

### `/help`
Lists all available commands with one-line descriptions. Useful at the start of any session or after forking.

### `/init-project`
One-time setup after forking the template repo. Asks the developer:
- Project name and description
- Tech stack (language, framework, database, test runner, linter)
- Developer preferences (coding style, commit format)

Then writes:
- `CLAUDE.md` with project-specific rules
- `memory/user_profile.md` with developer context
- `memory/preferences/coding_style.md` with stack-specific conventions
- `memory/preferences/testing_style.md` with test framework conventions

### `/prime`
Loads full project context into a fresh Claude session without reading every file. Reads:
- `CLAUDE.md`
- `memory/MEMORY.md` (index only — agents fetch specific files as needed)
- `PRD.md` (if it exists)
- Latest plan in `.agents/plans/` (if any)

Outputs a structured context summary so Claude knows the project state immediately.

### `/create-prd`
Interview mode — asks structured questions about the app to be built:
- What problem does it solve?
- Who are the users?
- What are the core features (MVP scope)?
- What is explicitly out of scope?
- What does success look like?

Uses a subagent to explore the existing codebase (if any) before asking questions, so context is not repeated. Writes the completed PRD to `.claude/PRD.md`.

### `/plan-feature [feature-name]`
Analyzes the existing codebase + PRD to generate a step-by-step phase plan. Uses subagents to:
- Analyze existing code patterns and structure
- Identify dependencies and risks
- Map out implementation steps with clear completion criteria
- Define what tests need to pass for each step

Writes the plan to `.agents/plans/phase-N-[feature-name].md`.

### `/execute [plan-path]`
Runs the full TDD development cycle for each step in the plan:

```
For each step in the plan:
  1. Read step description + acceptance criteria
  2. Write failing tests (pytest) — verify they fail before implementing
  3. Implement the feature to make tests pass
  4. Run tests → if failing, fix implementation → re-run (max 3 attempts)
  5. Run linter (from CLAUDE.md project config) → fix errors → re-run
  6. If all green → git commit with semantic message (feat/fix/test/refactor)
  7. Mark step complete in plan → advance to next step
  8. If unresolvable after 3 attempts → halt, report clearly to developer
```

The linter/formatter is read from `CLAUDE.md` so it's configurable per project (ruff, flake8+black, biome, etc.).

### `/test`
Standalone test runner. Runs the project's test suite, reports failures with context, attempts fixes, and re-runs. Does not commit. Used for spot-checking or debugging outside of `/execute`.

### `/commit`
Creates a semantic git commit following the project's commit style (from `memory/preferences/commit_style.md`). Generates a message from the staged changes using conventional commit format (feat/fix/docs/test/refactor).

### `/review`
Reviews a completed phase before marking it done:
- Checks test coverage against acceptance criteria in the plan
- Verifies alignment with PRD
- Runs linter
- Checks for obvious code smells or security issues
- Outputs a pass/fail report with specific items to address

## Memory System

Memory files are stable markdown files that persist across Claude sessions. They are **not** loaded automatically — they are loaded on demand via `/prime` or by agents that need specific context.

### Structure

| File | Purpose | Who writes it |
|---|---|---|
| `MEMORY.md` | Index of all memory files — one-line pointer per file | Developer + agents |
| `user_profile.md` | Developer role, background, preferences | `/init-project`, developer |
| `project_decisions.md` | Key technical decisions + rationale | Agents during `/execute`, developer |
| `feedback_corrections.md` | Past session learnings, what worked/didn't | Developer (manually after sessions) |
| `preferences/testing_style.md` | pytest patterns, fixture conventions, TDD discipline | `/init-project`, developer |
| `preferences/coding_style.md` | Naming conventions, file structure, error handling | `/init-project`, developer |
| `preferences/commit_style.md` | Commit message format and tags | `/init-project`, developer |

### Memory Workflow
1. **Session start** → `/prime` reads `MEMORY.md` index, loads pointers
2. **During execution** → agents fetch specific memory files as needed (e.g., `testing_style.md` before writing tests)
3. **After session** → agents or developer update `project_decisions.md` and `feedback_corrections.md` with learnings
4. **Preferences** → set once via `/init-project`, rarely changed

## Agent / Subagent Patterns

The harness uses subagent delegation to keep the main conversation context clean:

- **`/create-prd`** → spawns a subagent to explore the codebase before asking questions
- **`/plan-feature`** → spawns up to 3 parallel subagents: one for code analysis, one for dependency mapping, one for risk identification
- **`/execute`** → main agent runs the TDD loop; spawns subagent for lint fixing if errors are complex
- **`/review`** → spawns a subagent to run the code review independently (no shared state with implementation)

Subagents receive their context via explicit prompts (not memory files directly) — the orchestrating agent reads memory and passes relevant extracts to the subagent prompt.

## Development Workflow

```
Session Start
  → /prime            (load context from memory)

Requirements Phase
  → /create-prd       (generate PRD via structured Q&A)

Planning Phase  
  → /plan-feature     (generate phase plan from PRD)

Implementation Phase (repeat per phase)
  → /execute [plan]   (TDD cycle: tests → implement → lint → commit)
  → /review           (validate phase before advancing)

Between Sessions
  → update feedback_corrections.md with learnings
  → /prime on next session start
```

## Reusability (GitHub Template)

The harness ships with:
- All command files pre-written and ready to use
- Memory files with placeholder content and clear instructions
- `CLAUDE.md` template with sections to fill in
- `README.md` explaining the whole system

**To use for a new project:**
1. Fork the GitHub repo
2. Run `/init-project` → Claude asks questions and fills in project-specific details
3. Run `/create-prd` → plan the app
4. Run `/plan-feature` → create first phase plan
5. Run `/execute` → build the app with TDD

## Verification / How to Test the Harness

1. Fork the repo into a new directory
2. Run `/init-project` — verify CLAUDE.md and memory files are created correctly
3. Run `/prime` — verify context summary is accurate and concise
4. Run `/create-prd` with a simple app idea — verify PRD.md is written to `.claude/`
5. Run `/plan-feature` — verify phase plan appears in `.agents/plans/`
6. Run `/execute` on phase 1 — verify: tests written first, implementation follows, tests pass, lint passes, commit created
7. Check git log — verify semantic commit message was created
8. Check memory files — verify `project_decisions.md` was updated during the session
